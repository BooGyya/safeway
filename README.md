# SafeWay

교통약자 맞춤 안전 경로 추천 서비스

장애인, 노인, 휠체어 이용자, 임산부 등 교통약자가 안전하고 편리하게 이동할 수 있도록 돕는 웹 서비스입니다.

- 팀명: 소보로빵
- 단계: 13-PJT (AI 기반 추천 서비스)
- 기간: 2026.05.19 ~ 2026.06.26

## 팀원 정보 및 업무 분담

| --- | --- | --- |
| 전소연 | Backend | Django REST API 전체(회원·커뮤니티·경로·챗봇·관리자), 외부 API 연동(TMAP·카카오·공공데이터포털·OpenWeather·Solapi·GMS), 경로 탐색/안전점수 알고리즘, AI 챗봇 프롬프팅, LLM 콘텐츠 필터링, 공공데이터 수집·적재 |
| 김보경 | Frontend | Vue 3 UI 전체(지도·경로·커뮤니티·마이페이지·관리자), Kakao Maps SDK 연동, Pinia 상태 관리, 접근성 설정, 반응형 디자인, AI 챗봇 UI |

## 주요 기능

### 맞춤형 경로 추천
- 4가지 경로 제공: 기본 추천 / 계단 없는 경로 / 큰길 우선 / 날씨 반영
- 사용자 유형별 보행 속도 적용
- 음향신호기·잔여시간표시기 유무 기반 **안전 점수** 산출
- 경로 주변 병원·약국·복지시설 정보 표시
- 횡단보도 **실시간 보행신호 잔여시간** 표시 (서울시 V2X 연동)
- 즐겨찾기 및 탐색 기록 저장

### 지도
- 도보 / 대중교통 / 택시 이동 수단 선택
- 주변 시설 카테고리별 마커 표시 (병원, 약국, 복지시설, 엘리베이터)
- 서울 주요 지역 실시간 인구 혼잡도 표시
- 관리자 확인 위험구간 마커 표시

### 불편신고함 (커뮤니티)
- 위험·장애물·파손·공사 등 카테고리별 신고
- 위치 정보 및 사진 첨부
- 좋아요, 댓글, 팔로우 기능
- 관리자 신뢰도 부여 시 지도에 위험구간 반영
- AI 기반 부적절 콘텐츠 필터링

### 안심도우미 (AI 챗봇)
- 자연어 기반 주변 시설·경로 안내
- 현재 위치 기반 맞춤 답변
- 대화 내역 저장

### 마이페이지
- 사용자 유형 설정 (일반, 노인, 장애인, 휠체어, 임산부)
- 글꼴 크기 조절 (4단계)
- 보행 속도 설정
- SOS 긴급 문자 발송 (보호자 번호 등록)
- 즐겨찾기, 탐색 기록, 게시글, 팔로우 관리

### 관리자 페이지
- 게시글 관리 (신뢰도 부여, 위험구간 적용, 삭제)
- 사용자 관리 (활성화/비활성화, 삭제)
- AI 콘텐츠 필터링 모니터링
- 공지사항 관리 (작성, 수정, 삭제, 상단 고정)

## 추천 알고리즘에 대한 기술적 설명

### 안전 점수
횡단보도·위험구간·이동거리를 기준으로 1.0에서 감점하는 방식으로 0~1 사이 점수를 계산합니다 (화면에는 ×100으로 표시).

```
crossing_penalty = 횡단보도마다 0.015(음향신호기 있음) / 0.035(없음), 최대 -0.35
distance_penalty = (이동거리 / 500m) × 0.01, 최대 -0.25
danger_penalty   = 관리자 승인 위험구간 1건당 -0.15, 최대 -0.45

score = clamp(1.0 - crossing_penalty - distance_penalty - danger_penalty, 0, 1)
```
장거리 경로에서 감점이 무한히 누적되어 전부 0점이 되는 것을 막기 위해 항목별로 감점 상한을 두었습니다.

### 실시간 보행신호 잔여시간 (V2X)
서울시 V2X 데이터는 교차로 단위로 8방향(북/북동/동/남동/남/남서/서/북서) 잔여시간을 제공합니다. 횡단보도와 가장 가까운 V2X 교차로를 찾고, 교차로 중심점 → 횡단보도 좌표의 방위각을 계산해 8방향 중 하나로 매핑한 뒤 해당 방향 값만 사용합니다(다른 방향 값으로 대체하지 않아 차량신호 등 무관한 값이 섞이는 것을 방지). 값이 없으면 현재 차량신호 구간으로 보고 "대기 중"으로 표시합니다.

### AI 챗봇 추천
자연어 질의에서 위치·시설 키워드를 감지하면 DB와 카카오 API 검색 결과를 LLM 프롬프트에 주입해 자연어로 정리된 답변을 생성합니다. "~에서 ~까지" 패턴이 감지되면 경로 탐색 화면으로 자동 연동됩니다.

## 생성형 AI를 활용한 부분

GMS API(`gpt-5-nano`)를 두 기능에 적용했습니다.

- **AI 챗봇**: 매 요청마다 사용자 유형·보행속도·위치(역지오코딩)·실시간 날씨를 시스템 프롬프트에 동적으로 주입해 개인화된 답변 생성. 지하철 호선·환승처럼 실시간이라 추측하면 위험한 정보는 추측하지 않고 지도로 안내하도록 프롬프트에 명시(환각 방지). 최근 대화 10개를 컨텍스트로 주입해 멀티턴 대화 유지.
- **LLM 콘텐츠 필터링**: 게시글 작성 시 제목+내용을 LLM에 전달해 욕설/혐오/비방/선정적 표현 여부를 JSON 형식으로 강제 응답받아 자동 차단. `developer` 역할로 시스템 프롬프트를 분리하고 판단 이유까지 구조화해서 응답받음. 관리자가 신뢰도 낮은 게시글을 최대 50개까지 일괄 검사 가능.

## 브랜치 전략

```
master              ← 최종 배포용 (직접 push 금지)
└── develop         ← 통합 개발 브랜치
    ├── back        ← 백엔드
    └── front       ← 프론트엔드
```

### 작업 흐름

1. `back` 또는 `front` 브랜치에서 각자 작업
2. 작업 완료 후 `develop` 브랜치로 PR 요청
3. 코드 리뷰 후 머지
4. 최종 완성 시 `master` 머지

> 실제 진행 시에는 팀원이 2명인 점을 고려해, 매 기능 단위로 `back`/`front` → `develop` 머지를 빠르게 반복하고 `develop`이 안정화될 때마다 `master`로 머지하는 방식으로 운영했습니다. 별도 PR 리뷰 없이 직접 머지한 점은 초기 원칙과 다르게 진행된 부분입니다.

- 한국어로 작성

- 명령형으로 작성 (예: "구현했다" ❌ → "구현" ✅)
- 50자 이내로 간결하게

## 학습 내용 및 회고

- **공공데이터 정합성**: 출처마다 속성명·단위가 달라, 실제 서비스에 쓸 속성을 선별하고 표준화하는 작업이 필요했습니다.
- **외부 API의 숨겨진 단위/센티넬 값**: 서울시 V2X 신호 데이터는 문서상 "센티초 단위"였지만 실제로는 데시초(1/10초) 단위였고, 값이 없을 때를 나타내는 `36001`이라는 센티넬 값이 정상 값처럼 섞여 들어와 비정상적으로 큰 잔여시간이 표시되는 버그가 있었습니다. 원본 응답을 직접 비교해서 원인을 특정했습니다.
- **API 호출 한도 관리**: 실시간성을 높이기 위해 재조회 주기를 짧게 잡으면 외부 API의 1일 호출 한도(1,000회)를 금방 소진한다는 것을 직접 겪고, 정확도와 한도 소진 사이의 균형점을 찾았습니다.
- **향후 개선 방향**: AWS 등 클라우드 배포, 커뮤니티 샘플 데이터 fixture 추가.
